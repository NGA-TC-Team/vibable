## 왜 Vibable인가?

AI 코딩 도구는 강력하지만, 지시가 모호하면 엉뚱한 결과물을 내놓습니다. 대부분의 시행착오는 코드에서 생기는 게 아니라 **기획이 흐릿한 채로 프롬프트를 보내는 순간** 시작됩니다.

Vibable은 그 전 단계를 돕는 도구입니다. 무엇을 만들지, 누구를 위해 만들지, 어떤 화면이 필요한지를 체계적으로 정리하고 나서 AI에게 넘기면 — 첫 번째 결과물부터 훨씬 가까운 곳에서 시작할 수 있습니다.

> 복잡한 앱을 혼자 기획하거나, Claude Subagent · OpenClaw 같은 AI 에이전트를 설계할 때 모두 사용할 수 있습니다.

## 어떻게 작동하나요?

Vibable은 프로젝트를 **7단계 페이즈**로 구성합니다. 처음부터 끝까지 순서대로 채울 수도 있고, 필요한 페이즈만 골라 쓸 수도 있습니다. 프로젝트 유형에 따라 페이즈 구성이 달라집니다 — 웹·모바일 앱은 화면과 데이터 중심, CLI는 커맨드와 입출력 계약 중심, AI 에이전트는 행동과 안전 중심입니다.

### 앱 트랙 (웹 · 모바일)

| # | 페이즈 | 여기서 정리하는 것 |
|---|--------|-------------------|
| 1 | **기획 개요** | 이 프로젝트가 왜 필요한가? 누구를 위한가? 성공 기준은? |
| 2 | **유저 시나리오** | 실제 사용자는 어떤 상황에서 이 앱을 쓰는가? |
| 3 | **요구사항** | 반드시 있어야 할 기능과 있으면 좋은 기능을 구분 |
| 4 | **정보 구조** | 전체 화면 목록과 유저가 이동하는 동선 설계 |
| 5 | **화면 설계** | 각 화면의 목적, 목업 스케치, 상태별 UI |
| 6 | **데이터 모델** | 앱에 필요한 데이터와 그 관계 |
| 7 | **디자인 시스템** | 색상, 타이포, 컴포넌트 스타일 가이드 |

### CLI 트랙

사람 위주(`human-first`), 에이전트 위주(`agent-first`), 둘 다(`hybrid`) 세 가지 서브타입 중에서 고릅니다. 서브타입에 따라 기본 플래그·출력 포맷·보안 정책이 미리 세팅됩니다.

| # | 페이즈 | 여기서 정리하는 것 |
|---|--------|-------------------|
| 1 | **기획 개요** | 바이너리 이름, 배포 채널(Homebrew/npm/cargo …), 주 런타임 |
| 2 | **유저·에이전트 시나리오** | 사람/CI/AI 에이전트 중 누가 어떻게 호출하는가 |
| 3 | **CLI 요구사항** | OS·아키·셸 매트릭스, 파괴적 액션 정책, 인증 방식 |
| 4 | **커맨드 구조** | 루트 바이너리, 서브커맨드 트리, 전역·로컬 플래그, 컴플리션 |
| 5 | **입출력 계약** | stdin/stdout 포맷, exit 코드, 스트리밍, 머신 가독 JSON 스키마 |
| 6 | **설정 & 데이터** | config 파일·env·비밀 저장소·XDG 레이아웃 |
| 7 | **터미널 UX** | ANSI 팔레트, 로그 레벨, 도움말 템플릿, 에러 메시지, 에이전트 친화 체크리스트 |

### AI 에이전트 트랙 (Claude Subagent · OpenClaw)

AI 에이전트를 직접 설계할 때 사용합니다. 에이전트가 어떤 역할을 맡고, 어떤 도구를 쓰며, 어떤 상황에서 사람이 개입해야 하는지까지 단계별로 정리합니다.

| # | 페이즈 | 여기서 정리하는 것 |
|---|--------|-------------------|
| 1 | **기획 개요** | 이 에이전트가 왜 필요한가? 배포 환경은? |
| 2 | **유저 시나리오** | 사람·다른 에이전트·CI 중 누가 호출하는가 |
| 3 | **에이전트 요구사항** | 자율성 수준, 권한 경계, 컨텍스트 범위 |
| 4 | **에이전트 아키텍처** | Subagent 패턴(single / orchestrator / explore-plan-execute), 데이터 흐름 |
| 5 | **에이전트 행동 설계** | 시스템 프롬프트, 책임, 출력 형식 — OpenClaw는 soul·identity·heartbeat |
| 6 | **연동 & 도구 설계** | Tool 허용 목록, Hooks, MCP 서버 — OpenClaw는 채널·스킬·게이트웨이 |
| 7 | **안전 & 테스트 설계** | 위험 시나리오, Human-in-the-loop, 테스트 케이스, 롤백 |

## 주요 기능

**예제 프로젝트로 바로 시작**
첫 방문 시 7가지 예제 프로젝트가 자동으로 준비됩니다 — 웹앱, 모바일 앱, CLI 3종(human-first · agent-first · hybrid), Claude Subagent 팀, OpenClaw 개인 비서. 모두 7개 페이즈가 완전히 채워져 있어, 설명 문서 없이도 "어떻게 채우면 되는지"를 바로 감 잡을 수 있습니다.

**실시간 미리보기**
기획서를 채우는 동시에 오른쪽 패널에서 다이어그램과 목업이 그려집니다. 화면 구조는 자동으로 사이트맵 트리와 유저 플로우 다이어그램으로, CLI는 커맨드 트리와 터미널 샘플로 시각화됩니다.

**스케치 목업**
화면 설계 단계에서는 손그림 스타일의 와이어프레임 캔버스를 사용합니다. 모바일 · 태블릿 · 데스크톱 뷰를 각각 설계할 수 있고, 로딩 · 오프라인 · 에러 상태도 별도로 정의할 수 있습니다.

**아이디어 노트 (무한 캔버스)**
페이즈 작업과 별개로, 프로젝트마다 Milanote 스타일 무한 캔버스가 붙어 있습니다. 노트 · 이미지 · 링크 · 투두 · 표 · 도형 · 드로잉을 자유롭게 배치하고, 보드를 중첩해 아이디어 계층을 만들 수 있습니다. 이미지는 드래그·붙여넣기로 즉시 업로드되고, 작업은 자동 저장됩니다.

**정보 구조 품질 진단**
화면 구조를 채우면 Vibable이 자동으로 연결 오류, 목적 미입력, 도달 불가능한 단계 등 14가지 항목을 검사하고 품질 점수(0–100)를 보여줍니다.

**다양한 내보내기**
완성된 기획서는 JSON, PDF, Markdown, CLI 레퍼런스 문서, 또는 에이전트 바로 쓸 수 있는 ZIP 번들로 내보낼 수 있습니다.

**서버가 필요 없습니다**
모든 데이터는 브라우저(IndexedDB)에만 저장됩니다. 계정도, 인터넷 연결도 필요 없고, 오프라인에서도 그대로 동작합니다.

**URL 공유**
기획서를 링크 하나로 공유할 수 있습니다. 링크를 열면 상대방은 내용을 바로 볼 수 있습니다 (읽기 전용).

## 내보내기 옵션

| 형식 | 설명 | 사용 대상 |
|------|------|-----------|
| **JSON** | 전체 또는 현재 페이즈 데이터 | 백업, 다른 프로젝트에 재활용 |
| **PDF** | 전체 기획서를 보기 좋게 출력 | 팀 공유, 발표 |
| **Markdown** | 페이즈별 문서 | AI 프롬프트, 노션 등에 붙여넣기 |
| **DESIGN.md** | 디자인 시스템 가이드 문서 | 개발자 핸드오프 |
| **CLI Reference** | 커맨드 트리 · 플래그 · exit 코드 · 샘플을 합친 레퍼런스 문서 | CLI 도움말·man 페이지 초안 |
| **Claude Agent ZIP** | `.claude/agents/` 에 바로 쓸 수 있는 파일 묶음 | Claude Subagent 배포 |
| **OpenClaw ZIP** | SOUL · IDENTITY · TOOLS 등 워크스페이스 파일 묶음 | OpenClaw 배포 |

## 시작하기

### 설치

```bash
git clone https://github.com/NGA-TC-Team/vibable.git
cd vibable
bun install
bun run dev
```

브라우저에서 `http://localhost:3000`을 열면 바로 사용할 수 있습니다.

### 처음 사용한다면

첫 방문 시 워크스페이스에 **예제 프로젝트 7종이 자동으로 채워집니다.** 카드를 하나 열어보면, 7개 페이즈가 모두 어떻게 채워져 있어야 하는지 바로 볼 수 있습니다.

직접 만들 때는:

1. 워크스페이스에서 "새 프로젝트" 버튼을 눌러 프로젝트를 만드세요.
2. 프로젝트 유형을 선택합니다 — 웹 앱, 모바일 앱, CLI(서브타입 3종), 또는 AI 에이전트(Claude Subagent / OpenClaw).
3. 1페이즈부터 순서대로 채우거나, 이미 생각해둔 페이즈부터 시작하세요. 아이디어 노트 캔버스에 스케치부터 던져두고 시작해도 됩니다.
4. 오른쪽 미리보기 패널에서 내용이 어떻게 구조화되는지 실시간으로 확인하세요.
5. 다 됐으면 상단 내보내기 버튼으로 원하는 형식으로 받으세요.

## 기여하기

버그 리포트, 기능 제안, PR 모두 환영합니다.

1. 먼저 이슈를 열어 변경 사항을 논의해 주세요.
2. 순수 함수와 유틸에는 Vitest 테스트를, 주요 사용자 흐름에는 Playwright 테스트를 함께 작성해 주세요.
3. 함수형 스타일을 유지합니다 — 클래스 컴포넌트, 명령형 패턴은 사용하지 않습니다.
4. 런타임·패키지 매니저는 `bun` 기준입니다. `bun install` · `bun run dev` · `bun run test` · `bunx <tool>`을 사용해 주세요.
